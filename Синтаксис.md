# Оператор WITH (обобщённое табличное выражение)

**Обобщённое табличное выражение** или **CTE (Common Table Expressions)** - представляет собой временный именованный результат, который существует только в рамках одного SQL-запроса. Основное его назначение — улучшение читаемости запросов, а также повторное использование промежуточных результатов. Для написания обобщённого табличного выражения используется оператор `WITH`.

**MySQL**

```sql
-- Пример использования конструкции WITH
WITH Aeroflot_trips AS
    (SELECT TRIP.* FROM Company
        INNER JOIN Trip ON Trip.company = Company.id WHERE name = "Aeroflot")

SELECT plane, COUNT(plane) AS amount FROM Aeroflot_trips GROUP BY plane;
```

Выражение с `WITH` считается «временным», потому что результат не сохраняется где-либо на постоянной основе в схеме базы данных, а действует как временное представление, которое существует только на время выполнения запроса, то есть оно доступно только во время выполнения операторов `SELECT`, `INSERT`, `UPDATE`, `DELETE` или `MERGE`. Оно действительно только в том запросе, которому он принадлежит, что позволяет улучшить структуру запроса, не загрязняя глобальное пространство имён.

**Еще пример:**

``` sql
WITH CTE AS (
    SELECT EmployeeID, Salary
    FROM Employees
    WHERE Department = 'HR'
)
SELECT EmployeeID, Salary
FROM CTE
WHERE Salary > 50000;
```

**Особенности:**

- Существует только во время выполнения одного запроса.
- Не требует явного удаления.
- Можно использовать рекурсивные **CTE** для иерархических данных.

`WITH` используется для создания **CTE** (как в примере выше). Иногда его путают с временными таблицами, но он не создаёт физической таблицы в базе данных.

**Важно:** в некоторых СУБД (например, Oracle) оператор `WITH` может применяться для других целей, например, к материализованным видам (`WITH MATERIALIZED VIEW`).

## Синтаксис оператора WITH

**MySQL**

```sql
WITH название_cte [(столбец_1 [, столбец_2 ] …)] AS (подзапрос)
    , [название_cte [(столбец_1 [, столбец_2 ] …)] AS (подзапрос)] …
```

Порядок использования оператора WITH:

1. Ввести оператор `WITH`
2. Указать название обобщённого табличного выражения
3. Опционально: определить названия для столбцов получившегося табличного выражения, разделённых знаком запятой
4. Ввести `AS` и далее подзапрос, результат которого можно будет использовать в других частях SQL запроса, используя имя, определённое на 2 этапе
5. Опционально: если необходимо более одного табличного выражения, то ставится запятая и повторяются шаги 2-4

### Пример #2

Вот пример использования нескольких `WITH` для одного запроса. В этом случае каждое обобщённое табличное выражение (CTE) помогает разбить сложный запрос на логические шаги.

Допустим, у нас есть таблица `Sales`, содержащая данные о продажах:

``` sql
-- Таблица Sales
-- Columns: SaleID, ProductID, SaleDate, Amount, Price
WITH TotalSalesByProduct AS (
    SELECT 
        ProductID,
        SUM(Amount) AS TotalAmount,
        SUM(Amount * Price) AS TotalRevenue
    FROM Sales
    GROUP BY ProductID
),
TopProducts AS (
    SELECT 
        ProductID,
        TotalAmount,
        TotalRevenue
    FROM TotalSalesByProduct
    WHERE TotalAmount > 100
),
FinalResult AS (
    SELECT 
        TP.ProductID,
        TP.TotalAmount,
        TP.TotalRevenue,
        RANK() OVER (ORDER BY TP.TotalRevenue DESC) AS RevenueRank
    FROM TopProducts TP
)

SELECT 
    ProductID,
    TotalAmount,
    TotalRevenue,
    RevenueRank
FROM FinalResult
WHERE RevenueRank <= 5;
```

**Объяснение шагов:**

1. CTE `TotalSalesByProduct`: рассчитывает общий объём продаж (`TotalAmount`) и выручку (`TotalRevenue`) по каждому продукту.
2. CTE `TopProducts`: фильтрует продукты с объёмом продаж больше 100.
3. CTE `FinalResult`: добавляет ранжирование продуктов по выручке в порядке убывания.
4. Основной запрос: извлекает данные о топ-5 продуктах с наибольшей выручкой.

**Преимущества подхода:**

- Код структурирован и легко читается.
- Каждый `WITH` представляет собой отдельный шаг анализа.
- Упрощает поддержку и изменение запросов.

Запрос без использования `WITH` будет выглядеть более громоздким, так как промежуточные результаты нужно будет либо повторять, либо вкладывать друг в друга через подзапросы. Вот переписанный пример:

``` sql
SELECT 
    ProductID,
    TotalAmount,
    TotalRevenue,
    RevenueRank
FROM (
    SELECT 
        ProductID,
        TotalAmount,
        TotalRevenue,
        RANK() OVER (ORDER BY TotalRevenue DESC) AS RevenueRank
    FROM (
        SELECT 
            ProductID,
            SUM(Amount) AS TotalAmount,
            SUM(Amount * Price) AS TotalRevenue
        FROM Sales
        GROUP BY ProductID
        HAVING SUM(Amount) > 100
    ) AS FilteredProducts
) AS RankedProducts
WHERE RevenueRank <= 5;
```

**Разбор:**

1. Первый уровень вложенности (подзапрос `FilteredProducts`): рассчитывает общий объём продаж (`TotalAmount`) и выручку (`TotalRevenue`) по продукту. Отбрасывает продукты с общим объёмом продаж меньше или равным 100 (используется `HAVING`).
2. Второй уровень вложенности (подзапрос `RankedProducts`): добавляет ранжирование продуктов по выручке с использованием функции `RANK()`.
3. Внешний запрос: извлекает продукты, ранжированные в топ-5 по выручке.

**Сравнение с вариантом `WITH`**:

- Без `WITH`: Запрос более сложный для восприятия из-за вложенности. Логика шагов не так очевидна, и может быть сложнее внести изменения.
- С `WITH`: Логика разбита на отдельные шаги, каждый из которых проще читать, поддерживать и понимать.

Использование `WITH` в данном случае делает запрос более читаемым и модульным.

### Пример #3

Один из ключевых моментов, где `WITH` позволяет оптимизировать запрос, — это ситуация, когда одни и те же промежуточные данные используются несколько раз. Без `WITH` пришлось бы либо дублировать код, либо использовать временную таблицу.

Допустим, у нас есть таблица `Orders`, и мы хотим:

1. Вычислить общий объём заказов (`TotalOrders`) и сумму заказов (`TotalAmount`) для каждого клиента.
2. Использовать эти агрегированные данные для подсчёта:
    - Средней суммы заказа (`AverageOrderAmount`) для каждого клиента.
    - Общей доли клиента в общей сумме всех заказов (`CustomerShare`).
3. Сделать выборку клиентов, у которых объём заказов выше среднего по всем клиентам.

**Пример с использованием `WITH`:**

``` sql
WITH AggregatedOrders AS (
    SELECT 
        CustomerID,
        COUNT(OrderID) AS TotalOrders,
        SUM(OrderAmount) AS TotalAmount
    FROM Orders
    GROUP BY CustomerID
),
GlobalStats AS (
    SELECT 
        AVG(TotalOrders) AS AvgOrders,
        SUM(TotalAmount) AS TotalRevenue
    FROM AggregatedOrders
),
EnhancedData AS (
    SELECT 
        A.CustomerID,
        A.TotalOrders,
        A.TotalAmount,
        A.TotalAmount / G.TotalRevenue AS CustomerShare,
        A.TotalAmount / NULLIF(A.TotalOrders, 0) AS AverageOrderAmount
    FROM AggregatedOrders A
    CROSS JOIN GlobalStats G
)
SELECT 
    CustomerID,
    TotalOrders,
    TotalAmount,
    AverageOrderAmount,
    CustomerShare
FROM EnhancedData
WHERE TotalOrders > (SELECT AvgOrders FROM GlobalStats)
ORDER BY TotalAmount DESC;
```

**Объяснение:**

- CTE `AggregatedOrders`:
    - Агрегирует данные по каждому клиенту: количество заказов (`TotalOrders`) и их общая сумма (`TotalAmount`).
    - Эти данные используются в дальнейшем несколько раз для расчётов.
- CTE `GlobalStats`:
    - Рассчитывает глобальные статистики по всем клиентам:
        - Среднее количество заказов (`AvgOrders`).
        - Общую сумму всех заказов (`TotalRevenue`).
    - Эти данные тоже используются несколько раз для вычислений.
- CTE `EnhancedData`:
    - Обогащает данные о клиентах, добавляя:
        - Среднюю сумму заказа клиента (`AverageOrderAmount`), рассчитанную как отношение общей суммы заказов к их количеству.
        - Долю клиента в общей выручке (`CustomerShare`), которая вычисляется как отношение суммы заказов клиента к общей сумме всех заказов.
    - Для этого применяется `CROSS JOIN`, чтобы связать глобальные статистики из `GlobalStats` (одна строка с общей информацией) со всеми строками из `AggregatedOrders`.
        - Поскольку данные из `GlobalStats` одинаковы для всех клиентов, оператор `CROSS JOIN` позволяет объединить их без явного условия, распространяя глобальные метрики на каждый клиент.
- Основной запрос:
    - Выбирает клиентов, у которых количество заказов превышает среднее значение по всем клиентам (`AvgOrders`), используя данные из `GlobalStats`.
    - Сортирует клиентов по общей сумме заказов (`TotalAmount`) в порядке убывания, оставляя в финальном выводе только нужные столбцы.

> [!INFO]
> `CROSS JOIN` — это декартово произведение таблиц в SQL. Он объединяет каждую строку из первой таблицы с каждой строкой из второй таблицы, создавая все возможные комбинации строк.
> 
> Если у вас есть две таблицы:
>
> **Таблица A:**
> 
> |ID|Name|
> |---|---|
> |1|Alice|
> |2|Bob|
> 
> **Таблица B:**
> 
> |ProductID|ProductName|
> |---|---|
> |101|Laptop|
> |102|Phone|
> 
> После выполнения запроса с `CROSS JOIN`:
> 
> 
> ``` sql
> SELECT * FROM A CROSS JOIN B;
> ```
> 
> **Результат (декартово произведение):**
> 
> |ID|Name|ProductID|ProductName|
> |---|---|---|---|
> |1|Alice|101|Laptop|
> |1|Alice|102|Phone|
> |2|Bob|101|Laptop|
> |2|Bob|102|Phone|

**Как это выглядело бы без `WITH`:**

Без `WITH` весь запрос стал бы гораздо сложнее, с несколькими уровнями вложенных подзапросов, особенно учитывая необходимость повторного использования одних и тех же агрегированных данных. Вот пример, как это выглядело бы:

``` sql
SELECT 
    A.CustomerID,
    A.TotalOrders,
    A.TotalAmount,
    A.TotalAmount / (SELECT SUM(TotalAmount) FROM (
        SELECT SUM(OrderAmount) AS TotalAmount FROM Orders GROUP BY CustomerID
    ) AS GlobalRevenue) AS CustomerShare,
    A.TotalAmount / NULLIF(A.TotalOrders, 0) AS AverageOrderAmount
FROM (
    SELECT 
        CustomerID,
        COUNT(OrderID) AS TotalOrders,
        SUM(OrderAmount) AS TotalAmount
    FROM Orders
    GROUP BY CustomerID
) AS A
WHERE A.TotalOrders > (
    SELECT AVG(TotalOrders) 
    FROM (
        SELECT COUNT(OrderID) AS TotalOrders 
        FROM Orders 
        GROUP BY CustomerID
    ) AS AvgOrders
)
ORDER BY A.TotalAmount DESC;
```

**Сравнение:**

- С `WITH`: Код более читабелен, повторяющиеся вычисления (например, среднее значение и общая сумма) вынесены в отдельные именованные секции.
- Без `WITH`: Вложенные подзапросы усложняют чтение и сопровождение кода, повышают риск ошибок и снижают производительность.

**Оптимизация:**

СУБД (например, PostgreSQL или SQL Server) часто могут оптимизировать запросы с `WITH`, повторно используя результаты CTE вместо их повторного выполнения. Это делает подход с `WITH` не только читаемым, но и более эффективным.

## WITH RECURSIVE

В SQL `WITH RECURSIVE` используется для выполнения рекурсивных запросов, например, для работы с иерархическими данными, такими как деревья или графы.

**Где это полезно:**

1. Построение иерархий (например, сотрудников, категорий, файловых систем).
2. Вычисление путей в графах (например, маршрутов в транспортной сети).
3. Обход деревьев в реляционных базах данных.

### Пример #1. Работа с иерархией сотрудников (дерево)

Предположим, у нас есть таблица `Employees`:

| **EmployeeID** | **Name** | **ManagerID** |
| -------------- | -------- | ------------- |
| 1              | Alice    | NULL          |
| 2              | Bob      | 1             |
| 3              | Charlie  | 1             |
| 4              | David    | 2             |
| 5              | Eve      | 2             |
| 6              | Frank    | 3             |

**Задача:** построить дерево подчинённости сотрудников, начиная с менеджера `Alice`.

Запрос с `WITH RECURSIVE`:

``` sql
WITH RECURSIVE EmployeeHierarchy AS (
    -- Начальное условие (базовый случай): выбираем корневого менеджера
    SELECT 
        EmployeeID, 
        Name, 
        ManagerID, 
        1 AS Level
    FROM Employees
    WHERE ManagerID IS NULL -- Alice (корневой узел)

    UNION ALL

    -- Рекурсивный случай: находим подчинённых текущего уровня
    SELECT 
        e.EmployeeID, 
        e.Name, 
        e.ManagerID, 
        eh.Level + 1 AS Level
    FROM Employees e
    JOIN EmployeeHierarchy eh ON e.ManagerID = eh.EmployeeID
)
-- Основной запрос: выбираем результат
SELECT * 
FROM EmployeeHierarchy
ORDER BY Level, EmployeeID;
```

**Результат:**

| **EmployeeID** | **Name** | **ManagerID** | **Level** |
| -------------- | -------- | ------------- | --------- |
| 1              | Alice    | NULL          | 1         |
| 2              | Bob      | 1             | 2         |
| 3              | Charlie  | 1             | 2         |
| 4              | David    | 2             | 3         |
| 5              | Eve      | 2             | 3         |
| 6              | Frank    | 3             | 3         |

**Объяснение:**

1. *Базовый случай:*  
    Вначале выбираем корневого менеджера (`Alice`), у которого `ManagerID IS NULL`, и задаём начальный уровень `1`.
2. *Рекурсивный случай:*
    - Находим всех сотрудников, у которых `ManagerID` равен `EmployeeID` менеджера из предыдущего уровня.
    - При этом увеличиваем уровень (`Level`) на 1.
3. *Результат:*  
    Рекурсивно строится дерево сотрудников, начиная с менеджера и добавляя всех его подчинённых.

**Пример без `WITH RECURSIVE`**

Запрос без `WITH RECURSIVE` можно написать, но он будет гораздо сложнее для понимания и сопровождения, так как рекурсия должна быть реализована вручную через вложенные подзапросы. На практике, это менее эффективно и читаемо.

Допустим, мы знаем, что максимальная глубина иерархии — 3 уровня. Запрос можно написать следующим образом:

``` sql
-- Уровень 1 (корневой узел)
SELECT 
    e1.EmployeeID, 
    e1.Name, 
    e1.ManagerID, 
    1 AS Level
FROM Employees e1
WHERE e1.ManagerID IS NULL

UNION ALL

-- Уровень 2 (подчинённые корневого узла)
SELECT 
    e2.EmployeeID, 
    e2.Name, 
    e2.ManagerID, 
    2 AS Level
FROM Employees e2
JOIN (
    SELECT 
        e1.EmployeeID
    FROM Employees e1
    WHERE e1.ManagerID IS NULL
) AS Level1
ON e2.ManagerID = Level1.EmployeeID

UNION ALL

-- Уровень 3 (подчинённые подчинённых)
SELECT 
    e3.EmployeeID, 
    e3.Name, 
    e3.ManagerID, 
    3 AS Level
FROM Employees e3
JOIN (
    SELECT 
        e2.EmployeeID
    FROM Employees e2
    JOIN (
        SELECT 
            e1.EmployeeID
        FROM Employees e1
        WHERE e1.ManagerID IS NULL
    ) AS Level1
    ON e2.ManagerID = Level1.EmployeeID
) AS Level2
ON e3.ManagerID = Level2.EmployeeID
ORDER BY Level, EmployeeID;
```

**Результат:**

| **EmployeeID** | **Name** | **ManagerID** | **Level** |
| -------------- | -------- | ------------- | --------- |
| 1              | Alice    | NULL          | 1         |
| 2              | Bob      | 1             | 2         |
| 3              | Charlie  | 1             | 2         |
| 4              | David    | 2             | 3         |
| 5              | Eve      | 2             | 3         |
| 6              | Frank    | 3             | 3         |

**Объяснение:**

1. *Ручное деление на уровни:*
    - Мы вручную описываем выборки для каждого уровня. Уровень 1 выбирает корневые элементы (менеджеры без руководителя), уровень 2 — их подчинённых, уровень 3 — подчинённых второго уровня.
    - Для каждого следующего уровня мы должны заново использовать подзапросы, чтобы находить подчинённых предыдущего уровня.
2. *Нет автоматической рекурсии:*
    - Мы не можем динамически определять глубину вложенности. Если структура данных изменится (например, добавится уровень 4), нужно будет вручную добавлять ещё один `UNION ALL` с подзапросом.
3. *Сложность увеличивается экспоненциально:*
    - По мере увеличения глубины вложенности количество подзапросов и вложенных `JOIN` растёт, что делает запрос громоздким и менее эффективным.

**Основной недостаток подхода:**
- Если иерархия имеет неизвестную или большую глубину, этот метод становится непрактичным, так как требует заранее предсказать количество уровней.
- С использованием `WITH RECURSIVE` такая задача решается намного проще и динамичнее.

**Основной недостаток подхода:**

- Если иерархия имеет неизвестную или большую глубину, этот метод становится непрактичным, так как требует заранее предсказать количество уровней.
- С использованием `WITH RECURSIVE` такая задача решается намного проще и динамичнее.

### Пример #2. Поиск всех путей в графе

Давайте рассмотрим более сложный пример с использованием графов. Мы будем работать с маршрутом в транспортной сети, представленной в виде графа.

У нас есть таблица `Routes`, которая описывает связи между станциями:

| **FromStation** | **ToStation** | **Distance** |
| --------------- | ------------- | ------------ |
| A               | B             | 5            |
| A               | C             | 10           |
| B               | C             | 3            |
| B               | D             | 2            |
| C               | D             | 1            |
| D               | E             | 8            |

**Задача:** найти все возможные маршруты от станции `A` до станции `E` с указанием общей длины маршрута.

Запрос с `WITH RECURSIVE`:

``` sql
WITH RECURSIVE Paths AS (
    -- Базовый случай: начинаем с исходной станции A
    SELECT 
        FromStation,
        ToStation,
        Distance,
        CONCAT(FromStation, '->', ToStation) AS Path,
        Distance AS TotalDistance
    FROM Routes
    WHERE FromStation = 'A'

    UNION ALL

    -- Рекурсивный случай: добавляем новые станции к пути
    SELECT 
        p.FromStation,
        r.ToStation,
        r.Distance,
        CONCAT(p.Path, '->', r.ToStation) AS Path,
        p.TotalDistance + r.Distance AS TotalDistance
    FROM Paths p
    JOIN Routes r ON p.ToStation = r.FromStation
    WHERE r.ToStation NOT LIKE ANY (STRING_SPLIT(p.Path, '->')) -- Избегаем циклов
)
SELECT *
FROM Paths
WHERE ToStation = 'E' -- Оставляем только маршруты, которые заканчиваются на E
ORDER BY TotalDistance;
```

**Результат:**

| **FromStation** | **ToStation** | **Distance** | **Path**   | **TotalDistance** |
| --------------- | ------------- | ------------ | ---------- | ----------------- |
| A               | E             | 20           | A->B->D->E | 15                |
| A               | E             | 22           | A->C->D->E | 19                |

**Объяснение:**

1. *Базовый случай:*
    - Строим начальный маршрут, начинающийся с `FromStation = 'A'`.
2. *Рекурсивный случай:*
    - Для каждого маршрута добавляем новые станции, используя соединение между `ToStation` текущего маршрута и `FromStation` новой связи.
    - Чтобы избежать бесконечных циклов, проверяем, что новая станция не входит в уже построенный путь (`Path`).
3. *Фильтрация и сортировка:*
    - В итоговом запросе выбираем только маршруты, которые заканчиваются на станции `E`, и сортируем их по общей длине (`TotalDistance`).

**Сложные моменты:**

1. *Предотвращение циклов:*
    - Условие `NOT LIKE ANY (STRING_SPLIT(p.Path, '->'))` используется для проверки, чтобы станция не появлялась в пути дважды.
2. *Сбор пути:*
    - Используем `CONCAT` для создания текстового представления маршрута (например, `A->B->D->E`).
3. *Гибкость:*
    - Этот запрос универсален: добавление новых станций и маршрутов не требует изменения кода.

## Разница с временными таблицами

**Временные таблицы** — это физические таблицы, которые создаются в базе данных на время сессии или выполнения запроса. Они могут быть полезны, если промежуточные результаты используются в нескольких запросах или для больших объёмов данных.

**Пример создания временной таблицы:**

``` sql
CREATE TEMPORARY TABLE TempTable AS
SELECT EmployeeID, Salary
FROM Employees
WHERE Department = 'HR';

SELECT EmployeeID
FROM TempTable
WHERE Salary > 50000;

DROP TEMPORARY TABLE TempTable;
```

**Особенности:**

- Хранят данные физически на диске или в памяти.
- Могут использоваться в течение всей сессии (до явного удаления или её завершения).
- Требуют явного удаления, если это не сессионные таблицы.
- Удобны для работы с большими объёмами данных.

**Основные различия между CTE и временными таблицами:**

| **Характеристика**      | **CTE (WITH)**                              | **Временные таблицы**                       |
| ----------------------- | ------------------------------------------- | ------------------------------------------- |
| Срок существования      | Только в рамках одного запроса              | На время сессии (или до удаления)           |
| Использование ресурсов  | Не создаёт физической таблицы               | Создаёт физическую таблицу (память/диск)    |
| Масштабируемость        | Подходит для небольших промежуточных данных | Подходит для больших объёмов данных         |
| Повторное использование | Только в текущем запросе                    | В нескольких запросах в рамках одной сессии |
| Рекурсия                | Поддерживается (рекурсивные CTE)            | Не поддерживается напрямую                  |

Когда что использовать:

- Используйте **CTE** (`WITH`), когда требуется повысить читаемость сложного запроса или обработать промежуточные данные, не создавая физическую таблицу.
- Используйте **временные таблицы**, если данные нужно использовать в нескольких запросах или требуется обработка большого объёма данных.

## Заключение

Обобщённые табличные выражения были добавлены в SQL для упрощения сложных длинных запросов, особенно с множественными подзапросами. Их главная задача – улучшение читабельности, простоты написания запросов и их дальнейшей поддержки. Это происходит за счёт сокрытия больших и сложных запросов в созданные именованные выражения, которые потом используются в основном запросе.

Итог использования `WITH`:

- **Повторное использование данных:** данные, извлечённые в `WITH`, доступны для повторного использования, и СУБД, как правило, кэширует результат CTE. Это предотвращает лишние обращения к исходным данным.
- **Улучшение читаемости и поддержки:** запросы становятся более структурированными, так как промежуточные вычисления явно выделены.
